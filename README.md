# gravitino-semantic-layer-quickstart

A self-contained demonstration environment for the [Apache Gravitino](https://gravitino.apache.org/) semantic layer. It combines [dbt](https://www.getdbt.com/) and [MetricFlow](https://github.com/dbt-labs/metricflow) to compile a YAML-defined semantic model to SQL, and exposes those governed metrics to LLMs via a [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) server.

NYC yellow cab trip data (2024) is used as the demonstration dataset.

## What This Demonstrates

```
OSI YAML  →  MetricFlow (compiler)  →  SQL  →  DuckDB / Trino (execution)
                                                        ↑
                                              MCP Server (port 8003)
                                                        ↑
                                              LLM / MCP Client (Claude, etc.)
```

A semantic model defined in YAML declaratively specifies metrics, dimensions, and measures. MetricFlow compiles those definitions to optimized, ANSI-compatible SQL at query time — no hardcoded SQL anywhere. The MCP server exposes those compiled metrics as tools that LLMs can call by name.

This is the foundation of the Gravitino ADP (Agentic Data Protocol) vision: LLMs resolve natural language questions to governed metric SQL rather than querying raw tables directly.

## Prerequisites

- Python 3.12+
- NYC yellow cab Parquet files available locally (2024 dataset)
- Git

## Setup

### 1. Clone the repo

```bash
git clone https://github.com/markhoerth/gravitino-semantic-layer-quickstart.git
cd gravitino-semantic-layer-quickstart
```

### 2. Create and activate a virtual environment

```bash
python3 -m venv venv
source venv/bin/activate
```

You will need to run `source venv/bin/activate` each time you open a new terminal session.

### 3. Install dependencies

```bash
pip install dbt-core dbt-duckdb dbt-metricflow mcp
```

### 4. Configure your dbt profile

MetricFlow requires a dbt profile to locate your data warehouse connection. Create `~/.dbt/profiles.yml` if it does not already exist:

```yaml
gravitino_semantic_layer:
  target: dev
  outputs:
    dev:
      type: duckdb
      path: dev.duckdb
      threads: 1
    prod:
      type: duckdb
      path: prod.duckdb
      threads: 4
```

Then export the profiles directory so MetricFlow can find it:

```bash
export DBT_PROFILES_DIR=~/.dbt
```

Add this line to your `~/.bashrc` or `~/.zshrc` to make it permanent.

### 5. Set the Parquet data path

Edit `gravitino_semantic_layer/models/staging/stg_taxi_trips.sql` and update the path to your local NYC taxi Parquet files:

```sql
from read_parquet('/your/path/to/nyc_taxi_2024/*.parquet')
```

DuckDB reads the entire directory of Parquet files in one shot using the glob pattern.

### 6. Build the dbt models

```bash
cd gravitino_semantic_layer
dbt run
```

This creates two objects in DuckDB:

- `stg_taxi_trips` — a view over your local Parquet files
- `metricflow_time_spine` — a date table required by MetricFlow for time-based aggregations

### 7. Validate the semantic model

```bash
mf validate-configs
```

All checks should pass with zero errors.

## Querying Metrics Directly

MetricFlow compiles your YAML semantic model to SQL at query time. Use the `--explain` flag to see the generated SQL without executing it.

**Total trips by day**

```bash
mf query --metrics total_trips --group-by metric_time__day --explain
```

**Average fare by vendor**

```bash
mf query --metrics average_fare --group-by trip__vendor_id --explain
```

**List all available metrics**

```bash
mf list metrics
```

**List all available dimensions for a metric**

```bash
mf list dimensions --metrics total_trips
```

## MCP Server

The MCP server exposes governed metrics as tools that LLMs can call over HTTP. It wraps the MetricFlow CLI and serves responses on the [streamable-HTTP MCP transport](https://modelcontextprotocol.io/specification).

### Starting the Server

Use the provided convenience script from the repo root:

```bash
./start-metricflow-mcp.sh
```

Or start it manually with full environment variable control:

```bash
cd ~/gravitino-semantic-layer-quickstart
source venv/bin/activate

DBT_PROJECT_DIR=~/gravitino-semantic-layer-quickstart/gravitino_semantic_layer \
DBT_PROFILES_DIR=~/.dbt \
METRICFLOW_MCP_PORT=8003 \
python gravitino_semantic_layer/metricflow_mcp_server.py
```

The server listens on `http://0.0.0.0:8003/mcp` by default.

### Environment Variables

| Variable | Default | Description |
|---|---|---|
| `DBT_PROJECT_DIR` | `~/gravitino-semantic-layer-quickstart/gravitino_semantic_layer` | Path to the dbt project root |
| `DBT_PROFILES_DIR` | `~/.dbt` | Path to the directory containing `profiles.yml` |
| `METRICFLOW_MCP_PORT` | `8003` | Port the MCP server listens on |

### Available MCP Tools

**`list_metrics`**

Lists all governed metrics available in the semantic layer. Returns metric names, types, and descriptions. Use this tool first to discover what is available before querying.

**`query_metric`**

Compiles a named metric to SQL using MetricFlow. Returns the governed SQL that would be executed — not raw table access.

| Parameter | Required | Description |
|---|---|---|
| `metric` | Yes | Metric name (e.g. `total_trips`, `average_fare`) |
| `group_by` | No | Dimension to group by (e.g. `metric_time__day`, `trip__vendor_id`) |

### Connecting an MCP Client

To connect Claude Desktop to this server, add the following to your Claude Desktop MCP configuration:

```json
{
  "mcpServers": {
    "metricflow": {
      "url": "http://localhost:8003/mcp"
    }
  }
}
```

Once connected, Claude can call `list_metrics` and `query_metric` directly in conversation.

## Semantic Model

The semantic model is defined in `models/semantic_models/taxi_trips.yml`.

**Measures**

| Name | Aggregation | Description |
|---|---|---|
| `trip_count` | count | Total number of trips |
| `avg_fare` | average | Average fare amount |

**Metrics**

| Name | Label | Description |
|---|---|---|
| `total_trips` | Total Trips | Total number of taxi trips |
| `average_fare` | Average Fare | Average fare amount per trip |

**Dimensions**

| Name | Type | Notes |
|---|---|---|
| `pickup_datetime` | time | Day granularity; also available as `metric_time` |
| `vendor_id` | categorical | 1 = Creative Mobile Technologies, 2 = VeriFone |
| `payment_type` | categorical | 1 = credit card, 2 = cash, 3 = no charge, 4 = dispute |

## Relationship to Apache Gravitino

This quickstart is a standalone POC illustrating the semantic layer component of the Gravitino ADP architecture. The longer-term vision:

1. **Gravitino** provides a UI for authoring semantic models against catalog objects, generating OSI-compliant YAML
2. **MetricFlow** compiles the YAML to SQL at query time
3. **Trino** executes the SQL federally across heterogeneous data sources
4. **LLMs** access governed metrics via MCP — resolving natural language questions to trusted, compiled SQL rather than querying raw tables

This also contributes to the open design question under discussion in the OSI working group: whether a semantic model is a standalone catalog object or metadata enriching an existing Iceberg view.
